## 📌 1. Назначение

Данный документ содержит обязательные правила и рекомендации по работе с Git в рамках проекта.  
Он предназначен для всех разработчиков, участвующих в разработке, и служит единым источником информации.

В документе описаны:
- стандарты написания коммитов;
- структура и правила именования веток;
- стратегия GitFlow, адаптированная под проект;
- правила создания запросов на слияние веток;
- примеры и соглашения, обеспечивающие чистую историю репозитория.

Выполнение правил описанных в документе:
- обеспечивает единообразие в работе с Git;
- уменьшает количество конфликтов;
- упрощает поддержку истории проекта.

> ⚠️ Ознакомление с этим документом обязательно перед началом работы с кодовой базой.

## 🧾 2. Правила написания коммитов

Используйте [соглашение о коммитах (Conventional Commits)](https://www.conventionalcommits.org/en/v1.0.0/), 
чтобы коммиты были структурированными и понятными.

Структурированный коммит выглядит следующим образом:

```
<type>(<optional scope>): <description>

<optional body>

<optional footer(s)>
```

Формат **заголовка коммита**: `<тип>(<уточнение>): <описание>`, где уточнение — указание области изменений.

Настоятельно рекомендуются дополнительные пояснения в **теле коммита** если изменения сложные или требуются подробности.  

Если задача оформлена с помощью **GitLab Issues / GitHub Issues**, указывайте её номер в **футере коммита**, 
чтобы автоматически связать изменения с задачей. Например,

```
build(assets): настроена иерархия папки assets

Closes: #123
```

Допустимые метки в футере:
- `Fixes: #123` — автоматически закроет задачу при мёрже;
- `Closes: #123` — аналогично Fixes;
- `Refs: #123` — просто ссылается на задачу, без автоматического закрытия.

✅ Коммит должен быть:
- атомарным;
- в прошедшем времени (примеры глаголов: добавлен, реализован, устранена ошибка);
- в обезличенной форме (без "я", "сделал", "добавила");
- по существу и по делу;
- без точки в конце заголовка;
- длиной ≤ 72 символов;

🚫 Не допускается:
- использование нецензурной лексики или сарказма;
- личные комментарии или фразы типа "вроде заработало", "наконец-то", "ну не знаю, пускай будет";
- отладочные остатки без особой надобности (var_dump, console.log и т.д.).


### 🔤 2.1. Типы коммитов (type)

- `build` — изменения, касающиеся сборки проекта и внешних зависимостей;
- `ci` — настройка или изменение CI/CD;
- `chore` — рутинные правки, не влияющие ни на функциональность, ни на сборку: переименование файлов, уборка мусора и т.п.;
- `docs` — обновление или добавление документации (README, docstring, wiki и пр.);
- `feat` — новая функциональность;
- `fix` — исправление ошибок или некорректного поведения в коде;
- `perf` — изменения, направленные на улучшение производительности;
- `refactor` — рефакторинг кода без исправления ошибок или добавления нового функционала;
- `revert` — откат предыдущего коммита;
- `style` — изменения, связанные только с оформлением кода (отступы, форматирование, кавычки и т.д.);
- `test` — тесты, автотесты.


### 🗺️ 2.2. Область изменений (scope)

Scope — логический модуль/зона кода, где произошло изменение.

При оформлении scope используйте kebab-case, без пробелов, кратко.  

Разрешены вложенности с `/` или уточнения через `:`.  
Например: ui/element-object, db:migration

Когда указывать scope:
- ✅ Можно чётко назвать модуль/слой/директорию → указываем.
- ✅ Изменения затрагивают один основной модуль → указываем его.
- ❌ Изменения действительно "везде и понемногу" → допустимо опустить scope.

> ❗ Конкретные проектные scope подставляются в зависимости от модуля с которым ведётся работа.

> Для коммитов типа `docs` и `test` scope можно опускать.  
Если нужен — он должен быть содержательным: docs(readme), docs(workflow), test(unit) и т.д. (не docs(docs) и не test(tests)).


### 📎 2.3. Примеры корректных заголовков коммитов

```
build(zenject): добавлена библиотека zenject для работы с архитектурой проекта

chore(git): настроен .gitignore для игнорирования /build только в корне

docs(readme): обновлён README.md
docs(workflow): обновлён workflow.md

feat(auth): добавлен механизм авторизации через токены

fix(includes): устранён вызов функции *** вне среды ***

perf(TowerObject): оптимизирована запись TowerObject

refactor(GameManager): реализована вставка прочерков в таблицу с помощью цикла

style(GameManager): структурировано начало файла и упорядочены подключения
```

### ✏️ 2.4. Приведение коммитов к нужному виду

> ⚠️ Использовать `rebase -i` только в личных feature-ветках.
> После rebase изменятся хеши коммитов.

```bash
# Переключиться на ветку, в которой нужно поправить коммиты
git switch <тип/название-ветки>

# Обновить «слепки» удалённых веток
git fetch origin

# Интерактивно поправить сообщения/склеить коммиты (с сохранением даты)
git rebase -i --committer-date-is-author-date origin/develop
# В списке используйте: reword (переименовать), squash/fixup (склеить), drop (удалить)
# При конфликтах: исправить файлы → git add … → git rebase --continue
# Если патч пустой: git rebase --skip | Отменить: git rebase --abort

# Переписать удалённую ветку (без риска затереть чужие коммиты)
git push --force-with-lease
```

Быстрая правка последнего коммита (альтернатива `rebase -i`):
```bash
git commit --amend -m "type(scope): краткое описание"
```

## 🌿 3. Работа с ветками и GitFlow

> 💡 **Примечание:**  
> Проект использует расширенный кастомизированный GitFlow.


### 🔹 3.1. Основные ветки

| Ветка     | Назначение                                               |
|-----------|----------------------------------------------------------|
| `main`    | Стабильная версия, которая исполняется в продакшен-среде |
| `develop` | Основная ветка разработки. Объединяет все изменения      |


### 🔸 3.2. Поддерживающие ветки

| Ветка       | Откуда создаётся  | Куда мёржится      | Назначение                       |
|-------------|-------------------|--------------------|----------------------------------|
| `feature/*` | `develop`         | `develop`          | Новые фичи                       |
| `bugfix/*`  | `develop`         | `develop`          | Исправления до релиза            |
| `release/*` | `develop`         | `main` и `develop` | Подготовка и стабилизация релиза |
| `hotfix/*`  | `main`            | `main` и `develop` | Срочные исправления в релизе     |

> 💡 **Примечание:**  
> Допустимо использовать ветку `docs/*` для изменений, касающихся только документации (например, README, notes, схем и т.д.).  
> Такие ветки создаются от develop и мёржатся обратно в develop.  
> Используйте их, если изменения не связаны с кодом и не вписываются в другие ветки.  


### 🪄 3.3. Специальные (изолированные) ветки

| Ветка      | Назначение                                                                                       |
|------------|--------------------------------------------------------------------------------------------------|
| `orphan/*` | Изолированные ветки без родительской. Хранят SQL-запросы, документацию, схемы и служебные файлы. |

> 💡 **Примечание:**  
> Ветки `orphan/*` не мёржатся и не участвуют в разработке приложения.  
> Они используются как хранилище изолированных артефактов, которые не должны смешиваться с кодовой базой.


### 🖊️ 3.4. Правила именования веток

Всегда используйте **kebab-case** для части после префикса (`feature/`, `bugfix/`, `release/`, `hotfix/`, `docs/`).

Название должно кратко отражать назначение ветки.

Примеры:
- `feature/user-auth` — реализация авторизации
- `bugfix/login-error` — исправление ошибки входа
- `docs/readme-update` — правка документации


### ⚠️ 3.5. Слияние веток

**Все слияния выполняются только через Merge Request (MR) / Pull Request (PR)**.

**При слиянии рабочих веток необходимо всегда выполнять явный merge, а не fast-forward.**
Это позволяет сохранить в истории Git информацию о том, что изменения происходили в отдельной ветке, и делает структуру
коммитов более читаемой.

Чтобы предотвратить **fast-forward**, всегда используйте флаг `--no-ff`:

```bash
git merge --no-ff <тип/название-ветки>
```


### 🧹 3.6. Удаление веток после завершения работы

Чтобы не плодить мусор, удаляйте ветки после завершения работы с ними (если дальнейшая доработка не требуется).

```bash
git branch -d <тип/название-ветки>
git push origin --delete <тип/название-ветки>
```

Если MR / PR закрыт без слияния — ветка удаляется и локально, и на удалённом репозитории.

### 👥 3.8. Работа с ветками в команде

Каждый разработчик ведёт работу в своей `feature/`-ветке.  
Прямой пуш в `develop` и `main` запрещён — все изменения проходят через Pull Request (PR, GitHub) или Merge Request (MR, GitLab).

#### 3.8.1. Создание новой ветки

<u>Перед созданием **новой рабочей ветки**</u> необходимо обновить локальную копию родительской ветки (`develop` или `main`) до 
актуального состояния удалённого репозитория:

   ```bash
   # 1) Переключиться на локальный develop
   git switch develop
   
   # 2) Обновить локальный develop строго fast-forward’ом
   git pull --ff-only origin develop
   
   # 3) Создать новую фичу от обновлённого develop
   git switch -c <имя-локальной-ветки>
   ```

> ❗ Если при попытке обновить локальный develop вы получили отказ ("Not possible to fast-forward"), значит в ветке есть ваши коммиты.  

В таком случае вы можете "переписать" локальный develop:
   ```bash
   # 1) Обновить «слепки» удалённых веток
   git fetch origin
   
   # 2) Сбросить локальные коммиты
   git reset --hard origin/develop   
   ```

#### 3.8.2. Перебазирование (rebase) рабочей ветки

Перед созданием MR / PR рекомендуется обновить свою `feature/`-ветку, чтобы она находилась поверх актуального `develop`.
Это снижает вероятность конфликтов при слиянии:

   ```bash
   # 1) Обновить «слепки» удалённых веток
   git fetch origin
   
   # 2) Переключиться на свою рабочую ветку
   git switch <имя-локальной-ветки>
   
   # 3) Перебазировать её на свежий origin/develop
   git rebase origin/develop
   
   # 4) Проверить точку расхождения историй (коммит)
   git merge-base <имя-локальной-ветки> origin/develop
   
   # 5) Безопасно переписать свою ветку в удалённом репозитории
   git push --force-with-lease
   ```

> ⚠️ Rebase переписывает историю (включая хеши коммитов).
> Используйте только на личных feature-ветках, которые не используются другими разработчиками.

#### 3.8.3. Клонирование чужих веток (зеркало)

Если необходимо протестировать функционал из ветки другого разработчика, создаётся локальная копия («зеркало») удалённой ветки.
В такой ветке не делаются собственные коммиты — она используется только для просмотра и тестирования.

   ```bash
   # 1) Обновить «слепки» удалённых веток
   git fetch origin
   
   # 2) Создать локальную ветку, связанную с удалённой веткой другого разработчика
   git switch --track -c <имя-локальной-ветки> origin/<имя-удалённой-ветки>
   ```

Для последующего обновления зеркала при появлении новых коммитов в удалённой ветке:

   ```bash
   git switch <имя-локальной-ветки>
   git fetch origin
   git pull --ff-only
   ```

## 🛎️ 4. Запросы на слияние веток и взаимодействие

Один PR / MR = одна задача.  
Название PR / MR должно кратко отражать суть изменений.  

В описании указывается:
- цель задачи;
- ключевые изменения;
- важные замечания (например, временные заглушки).

Заголовок MR / PR должен быть:
- в прошедшем времени (примеры глаголов: добавлен, реализован, устранена ошибка);
- в обезличенной форме (без "я", "сделал", "добавила");
- по существу и по делу.


### ✅ 4.1. Чеклист перед отправкой PR / MR

- [ ] Код протестирован локально
- [ ] Удалён временный / отладочный код
- [ ] Коммиты приведены к нужному виду (cм. [примеры коммитов](#-23-примеры-корректных-заголовков-коммитов) и [правка коммитов](#-24-приведение-коммитов-к-нужному-виду))
- [ ] Подтянута актуальная версия из develop (сделано перебазирование рабочей ветки)
- [ ] Пройден ревью (если требуется)
